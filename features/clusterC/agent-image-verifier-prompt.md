## OTHER CONSIDERATIONS:

- **Sponsor-tech relevance: BOTH (heavy).**
  - **Gemini** — this is the *image-modality Gemini Pro vision story*. One vision call per profile; `gemini-2.5-pro` model; multimodal input with `IMAGE_BATCH_MAX` screenshots per call. The Gemini-Award narrative's *"Pro for the verifier agents (text + vision)"* (`features/architecture.md:62`) is *executed* here for the visual modality; the text-verifier sibling row owns the text half.
  - **Lobster Trap** — every `LlmClient.vision` call routes through Lobster Trap by construction (`@scout/llm-client.createLlmClient`'s baseURL points at the proxy per `PRPs/foundation-ad-verification.md:179`). The `_lobstertrap` declared-intent payload is the load-bearing piece for the image modality: an adversarial overlay-text attack that injects "ignore prior instructions" into a screenshot is detected by DPI as a declared-vs-detected mismatch on the *response* (the model's emitted classification doesn't match `declared_intent: "ad-verification.image-classification"`'s expected output shape), and Lobster Trap returns `verdict: "DENY"`. The verifier translates that into `AgentVerdict.decision: "DENY"` → arbiter consensus → profile DENY → gate honors → bid suppressed. This chain is the Veea-Award demo's image-modality story.

- **Open question — Batching strategy.**
  - **(A) One call per screenshot.** Simplest; verifier emits N separate `AgentVerdict`s and merges. Cost: N× Gemini Pro spend, N× latency (parallel under `Promise.all` ~= 1× latency but N× throughput cost on the warm path's rate limit). Lost context: the model can't compare across screenshots ("the gambling overlay is on screenshot 2 but the page header on screenshot 1 says 'news'" requires cross-image context).
  - **(B) Batched multimodal (recommend).** One call with all N screenshots as vision-message parts. Cost: 1× spend (with proportionally larger input-image tokens). Latency: ~3-6 s for 3 screenshots vs. ~3-6 s for 1 (Gemini's tokenization+inference scales sublinearly in image count). Context: the model sees the page as a sequence; cross-image reasoning is enabled.
  - **(C) Adaptive: start batched, fall back to per-screenshot on rate-limit (HTTP 429 / Lobster Trap `RATE_LIMIT`).** Best of both; complexity in the retry/fallback path; v2 work.
  - **Recommend (B) for v1, (C) as a follow-up filed in *Out of scope*.** (B) fits the typical 3-screenshot batch in one call; (A) wastes the cross-image context that distinguishes brand-safe news pages from brand-unsafe gambling overlays. The Q6 cost trip-wire (`features/clusterB/profiler-real-loop.md:99-103`) is the right place for adaptive degradation; this verifier accepts the `degradationHint` and responds via *transport switch* (inline → Files API), not *batching shrink*.

- **Open question — Image transport.**
  - **(A) Inline base64.** Simplest; one HTTP round-trip; bounded by Gemini's 20 MiB inline limit (`IMAGE_BYTES_MAX_INLINE = 5 MiB` per image is conservative). Adds ~33% payload overhead from base64 encoding.
  - **(B) Gemini Files API.** Two round-trips per file (upload, then reference) but no inline-size cap; better for >5 MiB images or repeated reference within a session. Higher latency on the upload leg.
  - **(C) URL fetched by Gemini.** Gemini dereferences an HTTP URL directly. Free for the verifier; but the harness emits `file://` URIs (unreachable by Gemini), AND letting the model fetch arbitrary URLs is an SSRF + prompt-injection vector. **Disqualifying.**
  - **Recommend (A) default, (B) on Q6 trip-wire degradation OR aggregate batch > `IMAGE_BATCH_INLINE_TOTAL_MAX` (12 MiB).** Inline is the v1 happy path. The Q6 trip-wire is the *transport switch trigger*: when window-cost crosses the soft limit, the profiler sets `ctx.degradationHint`; the image verifier reads it and switches to Files API to bound payload size (the cost saving is in not re-encoding the same image bytes inline on retries). `LlmClient.uploadFile` is a new method foundation/profiler PRPs must add — file as a coordination point.

- **Open question — Schema-bound output: image-region coordinates?**
  - **(A) Required: model emits `categories[].region: { screenshotIdx, x, y, w, h }` for every flagged label.** Best for the dashboard's evidence drill-down ("show me which pixels triggered the gambling classification"); doubles the prompt complexity (region-emission instructions); error-prone at the model side (Gemini's coordinate emission is unreliable per the vision docs); increases response token count.
  - **(B) Just `category` + `confidence` + `reason` (v1 default).** Verifier emits `categories: [{label, confidence}]` and `detectedEntities: [{name, type, confidence}]`. The `screenshot` `evidenceRef` cites which screenshot was implicated (whole image, not region). Dashboard renders the screenshot tile with the verifier's text reason as a caption.
  - **Recommend (B) for v1, (A) as a follow-up.** Mirror text-verifier's shape (no DOM-byte-offsets in v1) for arbiter cross-modality consistency. Region coords are an enhancement; the v1 demo doesn't need them.

- **Open question — Adversarial overlay text in screenshots.**
  - **(A) OCR + sanitize pre-LLM.** Run `tesseract.js` or a tiny vision-LLM round-trip over each screenshot, extract overlay text, sanitize for injection patterns ("ignore prior", "system:", role-confusion strings), drop matching screenshots before the vision call. Defense-in-depth; recursive cost (an LLM-based OCR is *itself* a vision call); `tesseract.js` is a ~3 MB browser/Node lib and slow.
  - **(B) Trust Gemini safety classifier + Lobster Trap DPI (v1 recommend).** No pre-LLM OCR. Gemini's own safety filter (`features/architecture.md` doesn't mention it but it's enabled by default on `generativelanguage.googleapis.com`) catches some adversarial prompts; Lobster Trap's DPI of the *response* catches declared-vs-detected intent mismatches on the model's output. Cheaper; latency-friendly; the demo's load-bearing answer.
  - **(C) Both.** A is layered on top of B for high-value advertisers; v2 work.
  - **Recommend (B) for v1, (C) as a productionization follow-up.** The Veea-Award narrative *requires* DPI to be the load-bearing defense — adding pre-LLM OCR steals the story from Lobster Trap. The adversarial-overlay fixture test (above) pins the defense's executability.

- **Q6 cost trip-wire integration.** The image verifier reads `ctx.degradationHint` (set by the profiler per `features/clusterB/profiler-real-loop.md:99-103`) and acts on `"drop_video"` → no behavior change here (that hint is for the video verifier; image is unaffected at the *what-to-classify* level) but DOES toggle to Files-API transport when the aggregate batch payload would exceed `IMAGE_BATCH_INLINE_TOTAL_MAX`. The `"collapse_text_image"` hint is NOT this PRP's responsibility — per `features/clusterB/profiler-real-loop.md:127` and the profiler's *Out of scope*, a separate `verifiers.combined` slot is filed for the collapsed text+image prompt; this verifier MUST NOT alone respond to `"collapse_text_image"` by inlining DOM text (that's the combined verifier's job — keep modality boundaries clean). Pin the no-op behavior on `"collapse_text_image"` in a test.

- **Lobster Trap seam preserved.** Explicit: every `LlmClient.vision` call goes through Lobster Trap by construction (foundation Q1 lock at `PRPs/foundation-ad-verification.md:151-154`). The image verifier package's ESLint config (inherited from root) blocks `openai` / `@google/genai` imports. **The `_lobstertrap` declared-intent payload is the load-bearing security property** — populated on EVERY call, never `undefined`, never `null`. Test asserts presence on every code path. The verifier does NOT pass advertiser identity into the model input (system message + user message text are advertiser-agnostic); identity stays in the `_lobstertrap` metadata where DPI consumes it but the model context window does not.

- **Fail-closed.** Reiterated: malformed model output → `HUMAN_REVIEW` synthesis with `confidence: 0` on every emitted category (NOT silent ALLOW). The "model returns decision outside enum" failure test pins this; flipping it to ALLOW must fail the test. Same asymmetry as `gate-verdict-logic.md:103`, `features/clusterC/agent-arbiter-scoring.md:179`.

- **Security guardrails:**
  - **No `GEMINI_API_KEY` in `packages/agents/image-verifier/**`.** Foundation rule (`PRPs/foundation-ad-verification.md:301`); only `@scout/llm-client/src/config.ts` reads it. The image verifier's config (if any) reads `IMAGE_BATCH_MAX`, `IMAGE_BYTES_MAX_INLINE`, `IMAGE_BATCH_INLINE_TOTAL_MAX` only.
  - **No advertiser identity in model context.** `ctx.advertiserId` / `ctx.policyId` go into `_lobstertrap` declared-intent, NOT into the user message text. A leaked advertiser ID in the prompt would (a) bias the model's classification on advertiser-correlated content, (b) expose advertiser identity to Gemini's training pipeline. Test asserts the user message body does NOT contain either string.
  - **No raw screenshot URIs logged.** Logger emissions structured-summarize (`{ url, contentHash, screenshotCount, rejectedCount, modelLatencyMs }`); never `console.log(screenshots)` or `logger.info({ capture })`. Same rule as `features/clusterB/profiler-real-loop.md:248`.
  - **Pre-LLM size/format gate is the SSRF defense.** A harness-emitted `file://` URI is trusted; an unexpected scheme (`http://`, `ftp://`, `gopher://`) is rejected pre-LLM AND NOT passed to `fs.promises.readFile` (which would reject anyway). The defense lives in both `batch.ts` (scheme check) and `transport.ts` (file-read guard).
  - **No cookie/session state, no global LLM client cache.** Each `verify` call uses the injected `deps.llm`; no module-level mutable state. Determinism property pinned by the determinism test above.
  - **Fail-closed on malformed input.** A `PageCapture` that fails `PageCaptureSchema.safeParse` (defense in depth — the profiler already parsed at commit time) → throw an error that the profiler classifies as a verifier failure, NOT a silent ALLOW.
  - **Bounded prompt size.** System message ≤ 1500 tokens (pinned by test); user message text portion ≤ 500 tokens; vision-message-part image count ≤ `IMAGE_BATCH_MAX = 6`. Prevents a hostile taxonomy-override (`ctx.taxonomyHint` with 10,000 labels) from blowing the context window.

- **Gotchas:**
  - **`gemini-2.5-pro` token-per-image cost.** Gemini charges per image based on resolution; a 1280×800 above-fold screenshot is ~1290 tokens. 3 screenshots ≈ 3870 image tokens + ~1500 system + ~500 user text + ~1000 response = ~7000 tokens/call. At Pro pricing this is ~$0.01/call. The Q6 cost trip-wire's `PROFILER_COST_WINDOW_SOFT` calibration (`features/clusterB/profiler-real-loop.md:276`) must account for this — file a coordination note.
  - **Gemini OpenAI compat layer is BETA** (`PRPs/foundation-ad-verification.md:215`). The vision-message format may change; the `response_format: {type: "json_object"}` support may regress. Wire health-check into the smoke script so a vendor-side schema break is caught before the demo.
  - **`response_format: { type: "json_object" }` on Gemini compat may not enforce shape, only "is JSON".** The model can return valid JSON that's missing required fields. `parse.ts`'s `safeParse` is what catches it — pin in the malformed-output test. Don't trust the compat layer's shape enforcement alone.
  - **Floating-point clamp on `confidence`.** Model emits `0.95001` → clamp to `0.95`? Or pass through? Pass through; clamp only on `< 0 || > 1`. Determinism: `Math.max(0, Math.min(1, x))` is float-deterministic.
  - **Model hallucinates labels outside taxonomy.** A taxonomy of `["adult_content", "gambling"]` does NOT prevent the model from emitting `categories: [{label: "war_zone", confidence: 0.7}]`. The post-response filter drops it. Test pins this.
  - **`Decision` enum vs. model output.** Model returns `"Deny"` or `"DENY!"` or `"deny"` — only uppercase exact `"DENY"` passes `DecisionSchema.safeParse`. Either (a) normalize case in `parse.ts` (recommend; `String(modelOutput.decision).toUpperCase().trim()`), or (b) instruct the prompt strictly. Do BOTH: the prompt's system message specifies the enum, and `parse.ts` normalizes case before parsing. A malformed casing should not round to `HUMAN_REVIEW` if the intent was clearly DENY.
  - **`AbortSignal` plumbing through `LlmClient.vision`.** `ctx.abortSignal` MUST be passed through to the SDK call so the per-verifier timeout (`PROFILER_VERIFIER_TIMEOUT_MS`) actually cancels the underlying HTTP request. If `LlmClient.vision`'s args don't accept a signal, file a coordination note for foundation/profiler PRP to add it. Without this, the timeout fires but the SDK call still runs to completion and bills tokens.
  - **Foundation's `image-verifier` stub will be upgraded by profiler-real-loop** to return a hardcoded valid `AgentVerdict` (per `features/clusterB/profiler-real-loop.md:145`). When this PRP lands, the upgrade is *overwritten* with the real prompt. Tests that assume verifier output varies by `capture.url` will silently pass against the stub and meaningfully assert only against this PRP's body. Make tests assert *interface shape and call patterns* against the stub, and *content* only with the real `LlmClient.vision` mock.
  - **Inline base64 of a 5 MiB PNG is 6.7 MiB on the wire.** With 3 screenshots that's ~20 MiB per call — under Gemini's 20 MiB inline limit but at the boundary. `IMAGE_BATCH_INLINE_TOTAL_MAX = 12 MiB` is conservative; trip-wire switches to Files API well before the cap.
  - **Empty `evidenceRefs[]` from the model is legal.** The model may emit a classification without per-image evidence. The verifier passes it through (does NOT fabricate refs); arbiter handles the empty case per `features/clusterC/agent-arbiter-scoring.md:65`.
  - **`capture.metadata.lang` may be `null`.** Pass through to the prompt as `"unknown"` (not the string `"null"`). Pin in test.
  - **`fs.promises.readFile` failures.** A `file://` URI that points to a missing file (race: harness emitted, then GC ran before verifier read) → reject the screenshot pre-LLM, same path as oversized. Do NOT throw the whole `verify` call; emit a warning and continue.
