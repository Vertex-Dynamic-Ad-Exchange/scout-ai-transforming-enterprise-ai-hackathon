  ```ts
  AuditStore.query(filter: {

    decision?: Decision,                   // ALLOW | DENY | HUMAN_REVIEW

  ```
  `AuditRow` is a discriminated union over `kind: "verdict" | "profile_job_dlq"`. The `verdict` variant carries `{ id, kind: "verdict", advertiserId, ts, request: BidVerificationRequest, verdict: VerificationVerdict, profile: PageProfile | null, declaredIntent: LobstertrapDeclaredIntent | null, detectedIntent: LobstertrapDetectedIntent | null }`. See *Other Considerations — Coordination with foundation*.
- **`@scout/dashboard-backend` (new package)** — a thin Fastify ESM app (mirrors `@scout/gate` runtime locks from `PRPs/foundation-ad-verification.md:25` so we don't add a new runtime for the dashboard). Three endpoints:
  - `GET /api/verdicts?since=&until=&decision=&pageUrl=&kind=&limit=&cursor=` → `{ rows: AuditRow[]; nextCursor }`. Returns ONLY rows for the authenticated advertiser; the route handler derives `advertiserId` from the session, never accepts it from the query string.
  - `GET /api/verdicts/:id` → `AuditRow` (404 if not owned by the authenticated advertiser; **404 — not 403** — so an adversary cannot enumerate which IDs exist for another advertiser, same enumeration-safe principle as `features/clusterA/gate-verdict-logic.md:102`).
  - `GET /api/evidence/:uri` → image proxy. `:uri` is a base64-url of the original `EvidenceRef.uri` (`packages/shared/src/schemas/profile.ts:17-19`); the backend validates the URI belongs to a row this advertiser owns (otherwise 404), then streams the bytes through. Prevents the client from being handed raw URIs that might leak structure (e.g., bucket names, signed URLs with cross-tenant validity).
  - All three routes behind a `preHandler` that asserts the session is valid; sessions are stub-auth in v1 (`x-advertiser-id` header against an in-memory allowlist seeded by `@scout/scripts.seedDashboardSessions`). Real auth is a follow-up.
- **Real-time updates** — see *Other Considerations* for the polling-vs-SSE-vs-WebSocket decision; recommendation is **polling at 1s** while the timeline view is mounted (cheap, demo-stable, no backpressure concerns; the SLA is "feels live on stage", not "every verdict in <100ms"). The polling interval is paused when the tab is hidden (`document.visibilityState`) and resumes on focus — bounded cost even with the dashboard left open overnight.

- **Open question — Real-time updates.**
  - **(A) Polling at 1s, paused when tab hidden, `ETag`/`If-None-Match` for cheap 304s.** Simplest; demo-stable; bounded cost; no server-side state. The "fresh verdict appears within 2s" SLA is met.
  - **(B) SSE from `/api/verdicts/stream`.** Lower latency on stage, but every connected client holds a long-lived TCP connection; the backend needs a fan-out abstraction; one stage demo doesn't justify it.
  - **(C) WebSocket.** Overkill — bidirectional is not needed (the client never pushes); same backpressure / fan-out concerns as SSE.
  - **Recommend (A).** The dashboard is on the warm-read path; the gate's SLA is unaffected. If the on-stage demo ends up with >10s of perceived lag (unlikely; AuditStore writes are fire-and-forget per `features/clusterA/gate-verdict-logic.md:97` and visible within ~1ms), revisit and add SSE as a v2 PRP — keep the option open by not coupling polling to the view code (the polling lives in the `@tanstack/react-query` config, replaceable with `useEventSource` without touching the view components).

- **Open question — Declared-vs-detected intent diff: where does "declared intent" live?**
  - **(A) Advertiser's policy.** Adds a new field to `PolicySchema` (e.g., `Policy.intent: { allowedCategories: [...], deniedCategories: [...] }`). Declared intent = "this advertiser will run iff page matches this profile."
  - **(B) Bid request.** Per-bid intent — every `BidVerificationRequest` carries a `declaredIntent: { creativeIntent: "promote financial services product" }` string. Per-bid granularity but a one-line free-text field is rarely useful.
  - **(C) Both — they CAN disagree, that's the point.** The advertiser's policy (long-lived, structured) is the declared intent for *the dashboard's diff view*; the LLM call's `_lobstertrap.declared_intent` (per-call, narrow — *"classify this page against this policy"*) is the declared intent for *Lobster Trap's DPI inspection*. The "detected intent" is what Lobster Trap actually observed in the prompt body (e.g., adversarial injection in DOM snippet). The Veea-Award diff is **declared at the advertiser-policy level vs. detected at the LLM-inspection level** — they can disagree when a page's content jailbreaks the verifier into seeing something the policy never intended.
  - **Recommend (C).** Schema extension: `Policy` gains an optional `Policy.declaredIntent: { description: string; allowedCategories: string[]; deniedCategories: string[] }` field in `packages/shared/src/schemas/policy.ts` (additive — every existing fixture remains valid). The `_lobstertrap.declared_intent` payload the verifier sends per LLM call cites the policy's `declaredIntent.description` verbatim; the `IntentDiff` view shows policy-declared on the left, Lobster-Trap-detected on the right. **The schema extension is filed as a follow-up to `policy-match-evaluation.md` or `gate-verdict-logic.md`** — this PRP does not block on it, falling back to an empty-state "declared intent not authored yet" panel when the field is absent.

- **Open question — Lobster Trap iframe contract.**
  - **(A) Iframe is separately-hosted by Lobster Trap.** The operator runs `./lobstertrap/serve.sh` (per `PRPs/foundation-ad-verification.md:283`) which exposes `http://localhost:8080/_lobstertrap/`. Our dashboard renders an `<iframe src={VITE_LOBSTERTRAP_URL}>` — Veea's UI loads in their own origin, our views render in ours, no shared DOM.
  - **(B) Render with a Lobster-Trap-provided URL per session.** Requires per-session bootstrapping handshake; nothing in Veea's docs (`https://github.com/veeainc/lobstertrap`) suggests this is a thing.
  - **(C) Lobster Trap exposes a JSON feed; we render our own UI for it.** Replaces the iframe with our component code — undoes foundation Q8's "Veea's UI directly is a Track-1 win" rationale (`features/architecture.md:170`).
  - **Recommend (A) — iframe pointed at the operator-run Lobster Trap server, `VITE_LOBSTERTRAP_URL` in `.env`.** This is foundation Q8's lock; preserve it. The iframe is **`sandbox="allow-same-origin allow-scripts"`** so Veea's UI can fetch its own audit data without our dashboard relaying it. **Do not** add `allow-top-navigation` — a compromised Lobster Trap UI must not be able to redirect the parent.

- **No secrets in client (HARD).** Per `CLAUDE.md § Working agreements` ("No `VITE_*` / `NEXT_PUBLIC_*` env var holds a secret"):
  - `GEMINI_API_KEY` is **only** read by `@scout/llm-client.config.ts` (`PRPs/foundation-ad-verification.md:209-213`). Neither `packages/dashboard/**` nor `packages/dashboard-backend/**` reads it. Verified by an automated test that imports the dashboard's source and asserts no string match on `GEMINI_API_KEY` or `OPENAI_API_KEY` in the built bundle.
  - **No raw policy payload reaches the client.** The drill-down shows `policyVersion` (a version string) and `Reason[]` (which cite *which rule* fired by `rule.id`, never `rule.match`'s literal pattern — same rule-string non-leakage as `features/clusterA/policy-match-evaluation.md:115`). An advertiser's `Policy.rules[].match` strings (the actual category/entity match terms) are advertiser-private and never serialized to the wire.
  - **No advertiser private fields beyond `advertiserId`.** The session header carries the advertiser ID; the backend looks up the row, returns only fields the dashboard explicitly renders (no spread of the full audit row). Server-side filtering, not client-side hiding.
  - **Evidence URIs are proxied, never raw.** `GET /api/evidence/:base64url` returns the bytes; the client never sees the underlying S3 URL / file path / signed URL — prevents leaking storage structure.
  - **No `_lobstertrap` content beyond the trace ID** at the wire level on the verdict response itself (already locked by `features/clusterA/gate-verdict-logic.md:104`). The `IntentDiff` view fetches the declared and detected intent through the backend, which authorizes them against the advertiser before serializing.

- **Tenant isolation (HARD).** Advertiser A cannot see advertiser B's verdicts.
  - The backend derives `advertiserId` from the session, **never** from a query param. A `?advertiserId=B` in the query is silently ignored — there is no code path that reads it.
  - The `AuditStore.query()` filter requires `advertiserId` (the read interface above marks it REQUIRED). A backend route that forgot to pass it would not type-check.
  - Cross-tenant `GET /api/verdicts/:id` returns **404**, not 403 (no enumeration). Tested in (B).
  - Cross-tenant evidence URI returns **404**. Tested in (B).
  - The Lobster Trap iframe is a *separate* admin surface — its tenant model is Veea's, not ours; the demo shows it but the dashboard treats it as opaque. **Do NOT** thread our `advertiserId` into the iframe URL (no `?as=A` parameter); we don't own that auth model.

- **Gotchas:**
  - **`VITE_*` env vars are shipped to the client bundle.** A future "let me put the Gemini key in `VITE_GEMINI_KEY` for the dashboard" optimization is the bug this rule prevents. The automated bundle-grep test above pins it.
  - **The Lobster Trap iframe's CSP may evolve.** If Veea ships a future Lobster Trap release with `frame-ancestors 'self'`, our iframe stops rendering. The fallback is a "Open Lobster Trap dashboard in new tab →" link adjacent to where the iframe would have rendered — implemented as the iframe's `onError` fallback. Test pins the link's presence.
  - **`@tanstack/react-query` polling can stampede on remount.** Set `refetchOnMount: false` for the timeline query when the cache is already fresh; `staleTime: 500ms` to deduplicate the 1s polling cadence against re-renders.
  - **`@tanstack/react-virtual` and unstable row heights.** Verdict rows are fixed-height by design (one-line URL via `text-overflow: ellipsis`, fixed badge size); variable-height rows break virtualization estimation. **Do not** wrap the URL — `white-space: nowrap` + ellipsis. The drill-down (variable-height) is rendered in its own pane and is not virtualized.
  - **`document.visibilityState` events fire on every tab switch.** Debounce the polling resume by 250ms — a user flipping between tabs shouldn't trigger a burst of fetches.
  - **The evidence proxy is a streaming endpoint.** Use `reply.raw.pipe()` on the Fastify side, not buffering — a 5 MB screenshot buffered in memory per concurrent request is a denial-of-service surface even on a demo machine.
  - **`HUMAN_REVIEW` rows can carry an empty `disagreements[]`** (e.g., the step-3 below-threshold escalation path per `features/clusterC/agent-arbiter-scoring.md:49`). The drill-down must render this case as "Confidence below threshold — see profile signals" rather than an empty list with no explanation.
  - **`AuditStore.query()` over a fresh-empty store returns `{ rows: [], nextCursor: null }`** — the timeline view's empty state must distinguish "no verdicts yet" from "fetch failed". Tested via the (A) edge case and the (A) failure case being two separate tests.
  - **The Lobster Trap iframe takes ~500ms to load** on a cold demo machine. The bottom-right pane shows a skeleton placeholder until the iframe `onLoad` fires; otherwise a judge sees the verdict views render instantly and the iframe "pop in" half a second later, which reads as broken.
  - **`axe-core` AA checks can false-positive on the projector palette.** Lock the chosen palette to a colorblind-safe and contrast-tested set; if `axe-core` flags it, document the palette + the WCAG citation in `packages/dashboard/src/theme.ts` rather than silencing the rule.
  - **The dashboard's bundle size matters for the demo.** Foundation ships no bundle-size budget; this PRP introduces `react` + `@tanstack/*` (~80 KB gzipped together). Document the size; reject any new dependency that would push it past 200 KB gzipped without explicit sign-off.
